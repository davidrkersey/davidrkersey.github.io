Feed Forward Regressor NN to Predict Medicare Payout¶

For my final project in my AI course, I built a feed forward regressor neural network to predict Medicare payouts to physicians and practices based on the Center for Medicare and Medicaid (CMS) dataset that I also analyzed in two previous notebooks.

My objective was to use the Keras package to iteratively and empirically develop a supervised, feed forward regressor neural network that learned to predict total annual Medicare payouts for medical practitioners and practices in the US based on numerous characteristics describing their businesses and patients. I accomplished this by building a model using Dense layers and experimenting with different kernel initializers, layer width, layer depths, optimizers and learning rate.

In the preliminary modeling, I achieved a test MSE of 0.074 with a non-pruned random forest regressor model. I set this MSE as my target MSE to beat with my neural network.

Summary of Final Model¶

My final achieved a MSE of 0.0709, which outperforms the random forest model. Additionally, it had 292,801 parameters, which follows the general rule of thumb of keeping parameters below number of instances. The low validation MSE suggests that the model is generalizable.

The model has three hidden Dense layers with widths 300, 500 and 250. All hidden layers use relu as their activation function. The output layer has one node and the activation function is linear to perform a regression. The optimizer is Adamax and the loss function is mse. The model runs on 75 epochs and a batch size of 400. It is also cross validated using 3-fold cross validation. I collected the MSE values at each epoch and averaged them across all 3 folds to assess the performance of my model. I call evaluate() to assess the performance of the model and summary() to describe the parameters.

Model Depth¶

I began model development by establishing a baseline model and experimenting with model depth. The baseline model had 2 shallow dense layers with 50 and 100 layers, respectively. The model was only trained on 20 epochs per K-fold CV and batch size of 500. The model trained in 1 minute and 43 seconds and produced a test MSE was 0.1518. After evaluating the plot of training and validation loss, it seemed that the model could benefit from increased depth and training time.

My next model increased layer depth five-fold, from 50 and 100 to 250 and 500, respectively. Additionally, epochs were increased from 20 to 35. This drastically increased the parameters (while staying under the 300k instance threshold) and reduced test MSE to 0.1032. There was still room to add more parameters and train the model even longer, however.

In my third model I increased layer depth from 250 and 500 to 350 and 600, respectively, increased epochs to 100 and reduced batch size from 500 to 100. This drastically increased training time from ~3 minutes to 41 minutes. It also increased the number of parameters from ~140K to ~230K. The test MSE was 0.0756 which was close to achieving my original goal. The training, validation loss plot indicated that there was perhaps too much traing time taking place as validation loss began creeping above training loss, indicating over-fitting.

I settled on moving forward with a model that trains on 50 epochs and batch size of 400.

Model Width and Optimizers¶

In the next model, I added another dense layer with 250 nodes and cut the number of nodes in the first two layers. The three layers had 300, 500 and 250 layers, respectively. This brought total parameters to ~292K which is still under my target threshold.

Additionally, I experimented with different optimizers. Up until now I had been using the Adam optimizer which has been empirically tested in the AI community and continually demonstrates good and fast results. The Adam optimizer combines the advantages of both the RMSprop and AdaGrad optimizers.

I used the GridSearchCV() function from SK-Learn to test my current model with 5 optimizers: SGD, RMSprop, Adam, Adamax, and Nadam. The GridSearchCV() results yielded a lowest test MSE for RMSprop, however, when I used RMSprop in my original script I yielded a very poor test MSE. This led me to try all 5 optimizers in my original script. Adamax scored the lowest with a test MSE of 0.0793, as compated to a test MSE of 0.08147 with Adam.

Incorporating Learning Rate¶

Learning rate controls how quickly the model adapts to a problem. A small learning rate will require more training epochs because smaller changes are made to the weights each update. A large learning rate will result in rapid changes but may cause the model to quickly converge on a suboptimal solution.

I tested numerous learning rates, starting at 0.01 and working down as small as 0.001. I compared validation MSE to evaluate model performance. Additionally, because I used small learning rates, I increased model training time from 50 epochs to 75 epochs and decreased the batch size to 400.

My model performed best with a learning rate of 0.0035. This yielded a test MSE of 0.0709 which beats the random forest model! I found that decreasing learning rate from 0.01 was useful until it led to over-fitting. For example, a learning rate of 0.001 yielded a test MSE of 0.1.

#create model
def create_model():
    model=Sequential()
    model.add(Dense(layer1_node, input_dim=X.shape[1], kernel_initializer = 'random_uniform',
                    bias_initializer = 'zero', activation='relu')) #First hidden layer
    model.add(Dense(layer2_node, activation='relu')) # Second hidden layer
    model.add(Dense(layer3_node, activation='relu')) # Third hidden layer
    model.add(Dense(1, activation='linear')) #Output layer. Must be linear activation for a regressor
    opt = Adamax(lr=0.0035) # Define optimizer with unique learning rate
    model.compile(optimizer=opt, loss='mean_squared_error', metrics = ['mse']) # Specify loss function
    return model

Conclusion¶

This model accomplishes the objective I set out to complete: beat the MSE of the random forest regression I performed. Although that margin is narrow, I learned about new optimizers, such as Adamax, and how to use learning rate. Fine tuning the learning rate hyperparameter allowed me to squeeze the validation MSE under the random forest regressor.

The model is complex with 292,801 parameters. Depending on the objective of deploying this model it might make more sense to limit the number of parameters. A high number of parameters tends to detract from generalizability. On the other hand, a validation MSE of 0.07 is pretty good and still suggests generalizability. Given the large dimensionality of the dataset, perhaps the model needed to be complex to extract meaning from all the features.
